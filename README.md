# opd-bot

Telegram-бот для автоматизации трудоустройства, практики и стажировки в ПТЗ.

Закрывает ≥70% рутинных операций HR: сбор документов от кандидатов, согласование дат собеседования/медосмотра/обучения, генерация кадровых документов (заявление на приём, направление на медосмотр, характеристика практики).

## Зачем этот бот

Раньше кандидат писал HR-специалисту в мессенджер, присылал сканы паспорта и СНИЛС, ждал ответа о дате собеседования, потом ещё раз — о дате медосмотра, потом HR вручную заполнял шаблон заявления в Word. Бот убирает это ручное посредничество:

- Кандидат ведёт себя с ботом как с онлайн-сервисом госуслуг — пошаговые вопросы, загрузка файлов, выбор даты из списка.
- HR не переключается между мессенджером, почтой и Word — всё происходит в одном интерфейсе того же Telegram.
- Документы формируются по шаблону в один клик: данные кандидата подставляются в заранее подготовленный `.docx`.
- Все действия сохраняются в базу — видно, кто, что и когда сделал.

## Как это выглядит для кандидата

Кандидат ничего не устанавливает, не заводит пароли и не ходит на внешние сайты. Он просто открывает Telegram и пишет `/start` нужному боту.

**Шаг 1. Представиться.** Бот спрашивает ФИО и телефон. Данные валидируются сразу (телефон — в формате `+79001234567`).

**Шаг 2. Выбрать цель.** Кнопки: «Трудоустройство», «Практика», «Стажировка». Дальше требования к документам подбираются автоматически под выбранную цель.

**Шаг 3. Загрузить документы.** Бот по одному просит нужные файлы — паспорт, СНИЛС, ИНН, направление от вуза и т.д. Проверяет тип файла (PDF, JPG, PNG) и размер. Если кандидат прислал что-то не то — сразу говорит, что исправить, без обращения к HR.

**Шаг 4. Записаться на собеседование и обучение.** Бот показывает список свободных слотов (дата и время), которые завёл HR. Кандидат выбирает удобный — место автоматически резервируется.

**Шаг 5. Следить за статусом.** В меню «Мои заявки» видно текущий этап: «Документы на проверке», «Принято на собеседование», «Одобрено» и т.д. При смене статуса приходит уведомление от бота.

**Связаться с HR.** Если что-то непонятно, в меню есть кнопка «Связаться с HR» — сообщение уйдёт ответственному сотруднику без пересылок и лишних контактов.

Главное меню кандидата меняется в зависимости от состояния заявки: пока заявки нет — видна кнопка «Подать заявку», как только заявка активна — появляются кнопки записи на слоты и изменения документов.

## Как это выглядит для HR

HR-сотрудник заходит в того же бота — после выдачи ему роли HR меню переключается на административное.

**Список заявок.** Все активные заявки со статусом, целью и временем подачи. Можно фильтровать по статусу («На проверке», «Ожидает собеседования», «Одобрено», ...) и по цели, искать по ФИО, @username или телефону.

**Карточка заявки.** Для каждого кандидата:
- кнопка «📄 Документы» — открывает список загруженных файлов, по каждому можно скачать, одобрить или отклонить с причиной (причина уходит кандидату);
- кнопки «📝 Сгенерированные» и «📄 Сформировать характеристику» — генерация заявления, направления на медосмотр и характеристики практики (см. [DOCX-шаблоны](#docx-шаблоны));
- «💬 Написать кандидату» — отправить произвольное сообщение через бота;
- «📎 Запросить доп. документ» — выбрать из каталога нужное требование и попросить кандидата доложить файл;
- «🗑 Отменить заявку» — административно закрыть заявку (не путать с отклонением кандидата по существу).

**Слоты.** Раздел «Слоты» — CRUD над датами собеседований и обучений. HR задаёт тип (собеседование/обучение), дату-время, продолжительность и вместимость. Слот виден кандидатам как вариант записи, пока не заполнится или пока HR не деактивирует его.

**Каталог документов.** Список требуемых документов по каждой цели. HR может добавить новое требование («Справка 086/у» для «Трудоустройство»), указать его код, допустимые mime-типы, максимальный размер — кандидаты с этой целью автоматически начнут получать запрос на этот документ.

**Генерация документов.** Для характеристики практики HR проходит короткий мини-мастер: вводит ФИО руководителя, тему, даты. Для заявления и направления поля заполняются из профиля кандидата автоматически. На выходе — готовый `.docx`, который HR скачивает и распечатывает.

**Отмена действия.** В любом HR-мастере (ввод дат слота, ввод темы практики, написание сообщения кандидату) доступна reply-кнопка «❌ Отмена» или команда `/cancel` — они корректно выходят из текущего шага и возвращают главное меню HR.

## Как это устроено на сервере

Бот — это один небольшой сервис, не SaaS и не кластер. Можно поднять на любом Linux-сервере с 1 CPU и 1 ГБ RAM.

### Топология

```
 Telegram-серверы  ◄───── HTTPS / long-poll ─────►  ваш сервер
                                                    ┌──────────────────────────┐
                                                    │ docker-compose:           │
                                                    │                           │
                                                    │  ┌────────────────┐      │
                                                    │  │ bot (Python)   │      │
                                                    │  │  aiogram 3.x   │      │
                                                    │  │  webhook/poll  │      │
                                                    │  └───────┬────────┘      │
                                                    │          │ asyncpg        │
                                                    │  ┌───────▼────────┐      │
                                                    │  │ postgres:16    │      │
                                                    │  └────────────────┘      │
                                                    │                           │
                                                    │  volumes:                 │
                                                    │   • ./storage  (файлы)    │
                                                    │   • ./templates (.docx)   │
                                                    │   • postgres_data         │
                                                    └──────────────────────────┘
```

Два контейнера: бот и Postgres. Сеть между ними — внутренняя docker-сеть, снаружи Postgres не торчит. Наружу бот общается только с серверами Telegram (`api.telegram.org`) по HTTPS.

### Два режима работы: long-poll и webhook

Бот умеет получать обновления от Telegram двумя способами — через *long-polling* (сам опрашивает Telegram) или через *webhook* (Telegram приходит на публичный HTTPS-URL).

| Режим | Когда использовать | Требования |
|---|---|---|
| **long-poll** (по умолчанию, `WEBHOOK_URL` пусто) | Dev, сервер за NAT, нет белого IP и SSL | — |
| **webhook** (`WEBHOOK_URL=https://...`) | Прод с белым IP и HTTPS | Открытый 443 порт, валидный SSL (или reverse-proxy: nginx/caddy/traefik) |

В webhook-режиме бот слушает на `WEBHOOK_HOST:WEBHOOK_PORT` (по умолчанию `0.0.0.0:8080`) — перед ним ставится reverse-proxy, который терминирует TLS и проксирует `/webhook` на бот.

### Что хранится и где

| Что | Где | Кто пишет |
|---|---|---|
| Пользователи, заявки, требования, слоты, логи аудита | Postgres (`opdbot` БД) | бот |
| Загруженные от кандидатов файлы | `./storage/<user_id>/<application_id>/<doc_kind>_<ts>.<ext>` | бот |
| Сгенерированные HR документы | `./storage/<user_id>/<application_id>/generated/<kind>_<ts>.docx` | бот |
| DOCX-шаблоны | `./templates/*.docx` | человек вручную |
| Логи бота | stdout контейнера → `docker compose logs bot` | loguru |
| Состояние FSM диалогов | Redis (если настроен) или память процесса | aiogram |

Если `REDIS_URL` не задан, FSM хранится в памяти процесса — это значит, что при перезапуске бота кандидаты, которые в этот момент проходили онбординг, потеряют прогресс и начнут с `/start`. Для прода рекомендуется поднять Redis.

### Что происходит при перезапуске

- Текущие открытые диалоги (FSM-состояния) теряются при рестарте, если не настроен Redis. Долгоживущие данные (заявки, документы) — в Postgres, они не теряются.
- Telegram сам ретранслирует недоставленные сообщения: бот при старте в poll-режиме забирает последние необработанные апдейты. В webhook-режиме Telegram повторит запрос до 24 часов.
- docker-compose настроен с `restart: unless-stopped` — если процесс упал, Docker его перезапустит.

### Нагрузка и масштаб

Целевая нагрузка — до ~500 активных кандидатов в месяц, десятки одновременных диалогов. Один инстанс бота с 1 vCPU справляется с этим без проблем; Postgres 16 с дефолтными настройками — тем более. Горизонтальное масштабирование (несколько инстансов бота) не поддерживается из-за локального `storage/` — при необходимости файлы надо переехать в S3-совместимое хранилище.

### Безопасность

- Токен бота и пароль Postgres — в `.env`, который не коммитится в git (см. `.gitignore`).
- `docker-compose.yml` требует, чтобы `POSTGRES_PASSWORD` был задан, иначе контейнеры не стартанут (syntax `${POSTGRES_PASSWORD:?...}`).
- Файлы кандидатов хранятся на локальном диске — убедитесь, что у сервера включён диск-шифрование на уровне ОС или провайдера.
- Telegram ID суперадминов задаётся в `SUPERADMIN_TG_IDS`. Только они могут выдавать роль HR через `/grant_hr`.
- Команда `/switch_role` (переключение роли между кандидатом и HR «на лету») доступна **только** при `DEV_MODE=true` в `.env`. В проде (`DEV_MODE=false` или переменная не задана) команда скрыта из списка `/` и на вызов отвечает «Команда доступна только в режиме разработчика». Используется исключительно для локальных/стейджинг-тестов, чтобы не заводить второй Telegram-аккаунт.
- Аудит-лог (таблица `audit_log`) фиксирует: кто выдал/забрал роль, кто одобрил/отклонил документ, кто отменил заявку.

## Быстрый старт (dev)

### 1. Установить зависимости

```bash
uv sync
```

### 2. Настроить окружение

```bash
cp .env.example .env
```

Открыть `.env` и заполнить:

```
BOT_TOKEN=<токен от @BotFather>
SUPERADMIN_TG_IDS=<ваш Telegram ID>
```

### 3. Создать таблицы и применить миграции

```bash
uv run alembic upgrade head
```

Это создаст `opdbot.db` с таблицами и заполнит справочники (цели, базовые требования к документам).

### 4. Запустить бота

```bash
uv run python -m opdbot.main
```

### 5. Тесты

```bash
uv run pytest
```

## Первоначальная настройка HR

После первого запуска бота (свежая база) роли `hr` нет ни у кого, кроме суперадминов из `SUPERADMIN_TG_IDS`. Чтобы выдать роль HR коллеге:

1. Коллега открывает бота и отправляет `/start` — это создаёт запись о пользователе в БД.
2. Узнайте его Telegram ID (через `@userinfobot` или из логов бота).
3. Суперадмин отправляет в бота:

   ```
   /grant_hr <telegram_id>
   ```

После этого пользователь видит HR-меню. Команда `/grant_hr` доступна только суперадминам — обычные HR её не видят.

Полный список зарегистрированных пользователей и их ролей доступен напрямую через БД:

```bash
sqlite3 opdbot.db "SELECT id, tg_id, full_name, role FROM users;"
```

или для Postgres:

```bash
docker compose exec postgres psql -U opdbot -c "SELECT id, tg_id, full_name, role FROM users;"
```

## Деплой (Docker)

```bash
# скопировать и заполнить .env
cp .env.example .env

docker compose up -d
```

`docker-compose.yml` поднимает PostgreSQL и бот-контейнер. Том `./storage` и `./templates` монтируются в контейнер. Контейнер бота перезапускается автоматически (`restart: unless-stopped`).

После первого запуска применить миграции:

```bash
docker compose exec bot uv run alembic upgrade head
```

Посмотреть логи:

```bash
docker compose logs -f bot
```

Обновить бот (после git pull):

```bash
docker compose build bot
docker compose up -d bot
docker compose exec bot uv run alembic upgrade head
```

## Бэкапы и восстановление

### Что бэкапить

1. **Postgres** — дамп БД (или сам файл `opdbot.db` в dev-режиме).
2. **`storage/`** — все файлы, загруженные кандидатами и сгенерированные HR.
3. **`.env`** — копию держать отдельно от сервера (пароли, токен).

Шаблоны (`templates/`) обычно хранятся в git; если они только на сервере — бэкапить тоже.

### Автоматические бэкапы

Готовые скрипты:

- Linux/macOS: `scripts/backup.sh`
- Windows: `scripts/backup.bat`

Скрипт складывает Postgres-дамп и архив `storage/` в `./backups/YYYYMMDD/`. Запуск через cron / Планировщик задач:

```cron
# ежедневно в 03:00
0 3 * * * /opt/opd-bot/scripts/backup.sh >> /var/log/opd-bot-backup.log 2>&1
```

### Ручной бэкап

```bash
# Postgres
docker compose exec -T postgres pg_dump -U opdbot opdbot > backups/db-$(date +%Y%m%d).sql

# Файлы кандидатов
tar -czf backups/storage-$(date +%Y%m%d).tar.gz storage/
```

### Восстановление

```bash
# БД
cat backups/db-20260418.sql | docker compose exec -T postgres psql -U opdbot -d opdbot

# Файлы
tar -xzf backups/storage-20260418.tar.gz
```

## Healthcheck и мониторинг

**Docker healthcheck.** Контейнер бота имеет встроенный healthcheck (`docker-compose.yml`), который проверяет, что процесс жив. При `docker compose ps` статус контейнера показывается как `healthy` / `unhealthy`.

**HTTP /health.** При работе через webhook (`WEBHOOK_URL` задан) бот поднимает дополнительный эндпоинт `GET /health`, возвращающий `{"status": "ok"}` и код 200. Его удобно вешать на uptime-проверки (UptimeRobot, Healthchecks.io, Prometheus blackbox exporter).

**Логи.** loguru пишет в stdout контейнера. Для прода рекомендуется настроить сборщик (Loki, ELK, syslog) или хотя бы логротацию Docker:

```yaml
# docker-compose.yml — добавить в сервис bot
logging:
  driver: "json-file"
  options:
    max-size: "10m"
    max-file: "5"
```

**Что смотреть при проблемах.** Типичные симптомы и куда идти:

| Симптом | Где смотреть |
|---|---|
| Бот не отвечает на `/start` | `docker compose logs bot` — ошибки подключения к Telegram или БД |
| Кандидаты теряют прогресс при обновлении | Не настроен `REDIS_URL`, FSM в памяти |
| HR не видит сгенерированные файлы | Проверить права на `./storage/` и том в docker |
| `POSTGRES_PASSWORD must be set` при старте | Не заполнен `.env` |
| Webhook не работает | reverse-proxy/SSL, открыт ли 443, правильный ли `WEBHOOK_URL` |

## Разработка

```bash
# установить зависимости (включая dev)
uv sync

# запустить тесты
uv run pytest

# создать новую миграцию после изменения моделей
uv run alembic revision --autogenerate -m "описание"

# применить миграции
uv run alembic upgrade head

# откатить последнюю миграцию
uv run alembic downgrade -1
```

Также есть `Makefile` с сокращениями: `make install`, `make migrate`, `make dev`, `make test`, `make lint`, `make fmt`, `make typecheck`.
